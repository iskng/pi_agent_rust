`src/agent.rs:1668-1673`, `src/agent.rs:1777-1784`, `src/agent.rs:1994-1998`

Both the parallel and sequential abort paths mark a tool as started before `execute_tool()` reaches `ToolExecutionStart`. When extensions are enabled, `execute_tool()` awaits `dispatch_tool_call_hook(...)` first, so an abort during that hook window produces `ToolExecutionEnd { executed: true }` even though no `ToolExecutionStart` was emitted and no host adapter ever ran. That breaks the new lifecycle contract for any extension-enabled session aborted while tool-call hooks are still in flight.

Suggested fix: only set the started/executed bookkeeping from the same point that emits `ToolExecutionStart`, or thread an explicit "adapter entered" signal back out of `execute_tool()` instead of pre-marking the index before the hook path completes.

## Resolution
Moved executed/start bookkeeping onto the exact ToolExecutionStart boundary so extension-hook aborts before adapter entry now emit executed=false and no ToolExecutionStart.
