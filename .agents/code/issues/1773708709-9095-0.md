`EventBridge::handle_agent_event` maps every `AgentEvent::ToolExecutionStart` to `EmbedEvent::ToolStarted` (`crates/pi_lynx_sdk/src/event_bridge.rs:117-126`), but the core agent emits `ToolExecutionStart` for *all* tool calls before any host adapter begins running (`src/agent.rs:1727-1733`). In interrupted multi-tool batches or steering-induced skips, callback-only hosts therefore receive normal `ToolStarted`/`ToolCompleted` lifecycle events for tools whose adapters never actually ran, even though `executed=false` is already available on the corresponding `ToolExecutionEnd` event.

Suggested fix: either defer `EmbedEvent::ToolStarted` until real adapter entry is known, or extend the embed event contract so skipped/aborted tool calls carry explicit execution state instead of looking identical to successfully started tools.

## Resolution
Moved ToolExecutionStart to actual tool adapter entry in src/agent.rs and threaded EmbedEvent::ToolCompleted.executed so skipped or aborted synthetic completions are distinguishable without emitting false start events.
