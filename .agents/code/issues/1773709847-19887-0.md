`src/agent.rs:1677-1694`

`execute_parallel_batch()` races the entire `buffer_unordered(...).collect::<Vec<_>>()` future against the abort signal and returns `Vec::new()` on abort. In a read-only multi-tool batch, that drops outputs from tools that already finished before the abort fired, so phase 3 rewrites those completed tools into synthetic "Tool execution aborted" results instead of preserving their real output. This is observable for any turn that issues multiple read-only tools where one finishes quickly and another is still running when the host/user aborts.

Suggested fix: accumulate completed parallel results incrementally while the batch is running and return the finished subset on abort, rather than discarding the entire partial collection when `abort_fut` wins.

## Resolution
Preserved completed read-only batch results on abort by collecting buffered parallel tool completions incrementally instead of dropping the whole batch when the abort wins.
