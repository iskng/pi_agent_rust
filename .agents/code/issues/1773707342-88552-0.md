Problem: `EventBridge::handle_agent_event()` increments `tool_calls_executed` for every `AgentEvent::ToolExecutionEnd`, but `Agent::execute_tool_calls()` emits synthetic `ToolExecutionEnd` events for tool calls that never entered a host adapter when a multi-tool batch is interrupted (for example, aborting during the first tool still produces aborted end events for later calls in the batch).

Why this matters: `TurnResultMetadata.tool_calls_executed` is documented as the number of host tools executed during the turn. In an interrupted multi-tool batch it now over-reports execution, so Lynx cannot distinguish "one tool ran, one was aborted before execution" from "both tools ran" when it persists or audits the completed turn.

File: crates/pi_lynx_sdk/src/event_bridge.rs:141-149
Affected path: src/agent.rs:1700-1707,1857-1886

Suggested fix: count only tool calls that actually enter the adapter execution path instead of every `ToolExecutionEnd`, or carry an explicit executed/aborted-before-start bit through the bridge so synthetic abort completions do not increment `tool_calls_executed`.

## Resolution
Added an explicit started/executed bit to tool-end events, counted only executed tool ends in the Lynx event bridge, and added multi-tool abort regression coverage so synthetic aborted completions no longer inflate tool_calls_executed.
